项目概览

Pi（π）是一个由 Mario Zechner 创建的极简终端编码代理框架，托管在 Earendil Works 组织下。目标是构建一个核心紧凑、可高度扩展的编码代理。与 Claude Code、Codex、Amp 等现有工具不同，Pi 刻意保持小体量，提供精确的上下文工程控制，并且拥有超小系统提示（<1000 tokens）。

核心包详解

1. pi-ai — 统一 LLM API

pi-ai 是一个轻量级、多供应商的 LLM 抽象库，提供了统一的接口来访问不同的大语言模型供应商。它只包含四种核心 API：文本流、完整响应、图片生成和嵌入。

🌐 多供应商支持

原生适配 Anthropic Messages API、OpenAI Completions/Responses API、Google Generative AI API。兼容任何 OpenAI-compatible 端点（vLLM、Ollama、llama.cpp、LM Studio 等）。

🔄 跨供应商上下文切换

设计之初就支持在会话中无缝切换模型/供应商。将 Anthropic 的思维链转换为 <thinking> 标签内容块，使另一模型能理解上下文。

🧰 TypeBox 工具定义

使用 TypeBox schema 定义工具参数，自动验证并在失败时返回详细错误给模型重试。工具结果可拆分：一部分给 LLM 阅读，一部分给 UI 渲染。

🛑 完整的取消支持

使用 AbortController 在整个管道中支持取消，包括工具调用。取消后仍可获取部分结果。

pi-ai 的四大 API：

API说明
stream()流式响应，逐 token 输出事件（text_delta、tool_call、tool_result、error、done）
complete()完整响应，等待最终结果返回
generateImages()图片生成，一次性 API
embed()文本嵌入

模型注册表自动从 OpenRouter 和 models.dev 解析，包含 token 成本和能力信息（图片输入、推理支持等）。未注册的模型可手动定义：

const ollamaModel: Model<'openai-completions'> = {
  id: 'llama-3.1-8b',
  name: 'Llama 3.1 8B (Ollama)',
  api: 'openai-completions',
  provider: 'ollama',
  baseUrl: 'http://localhost:11434/v1',
  reasoning: false,
  cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
  contextWindow: 128000,
};

2. pi-agent-core — 代理运行时

pi-agent-core 在 pi-ai 之上构建了完整的代理循环（Agent Loop），处理工具执行、状态管理和事件流。

🔄 代理循环

处理完整的编排流程：处理用户消息 → 执行工具调用 → 将结果反馈给 LLM → 重复，直到模型产生不含工具调用的最终响应。

📨 消息队列

每次模型响应后，通过回调检查待处理消息并注入。支持单条注入和批量注入两种模式。

🔌 传输抽象

提供传输层抽象，支持直接运行代理或通过代理（proxy）运行，灵活性高。

📎 附件处理

内置图片和文档的附件处理能力，自动转换为目标供应商的原生格式。

3. pi-tui — 差分渲染终端 UI

pi-tui 是一个自建的轻量级 TUI 框架，采用保留模式（Retained Mode）和差分渲染（Differential Rendering）实现近乎无闪烁的终端体验。

工作原理

组件是一个包含 render(width) 方法和可选的 handleInput(data) 方法的对象。Container 垂直排列子组件并收集它们的渲染行。TUI 类维持一个"后缓冲"，记录之前写入终端的内容。每次更新时，与当前渲染结果逐行比较，只重写差异部分。

差分渲染算法

首次渲染： 所有行直接输出到终端。
宽度变化： 完全清除屏幕并重新渲染（软换行变化）。
正常更新： 找到与屏幕内容不同的第一行，将光标移到该行，从该行开始重新渲染到末尾。
视口外变化： 如果第一个变化行在当前可见视口上方（用户已向上滚动），则执行完全清除和重新渲染。

防闪烁

所有渲染操作包裹在同步输出转义序列（CSI ?2026h / CSI ?2026l）中，告诉终端缓冲所有输出并原子化显示。在 Ghostty、iTerm2 等现代终端中完全无闪烁。

4. pi-coding-agent — CLI 编码代理

这是终端用户直接使用的 CLI 工具。将所有组件组合在一起，提供完整的编码代理体验。

💻 四模式运行

交互式终端模式、print 模式（输出到 stdout）、JSON 模式（结构化输出）、RPC 模式（进程集成）。

📂 会话管理

支持 continue、resume 与会话分支（从历史消息回退并创建新分支）。HTML 导出功能。

📝 AGENTS.md

支持全局和项目级别的 AGENTS.md 文件，分层加载到系统提示中。可自定义或完全替换系统提示。

🎨 扩展系统

TypeScript 扩展、Skills（技能）、提示模板（自定义斜杠命令）、主题（可热重载）、Pi 包（npm 分发）。

Pi CLI 的核心工具集仅 4 个（另有一些只读工具可选），系统提示 + 工具定义合计 不到 1000 tokens：

工具功能
read读取文件内容（文本和图片），支持 offset/limit
write写入文件，自动创建父目录
edit精确替换文件内容（基于旧文本匹配）
bash执行 bash 命令，可选超时

设计哲学

Pi 在设计和实现上与主流编码代理有着显著的不同之处。

🎯 极简系统提示

Pi 的系统提示不到 1000 tokens，远少于 Claude Code 的数万 tokens。核心信念是：前沿模型经过充分的 RL 训练，从根本上理解编码代理的任务，不需要冗长的提示工程。Terminal-Bench 基准测试表明，极简提示在性能上并不输于巨型提示。

🦅 YOLO 模式（默认）

Pi 默认具有完全的 filesystem 访问权限和任意命令执行权限，没有权限检查和安全护栏。Pi 团队认为：一旦代理能写代码和执行代码，安全措施基本上是"安全剧场"。真正的解决方案是在容器中运行，而不是用假护栏麻痹用户。Simon Willison 提出的"双 LLM 模式"也承认引入巨大的实现复杂性却无法根本解决问题。

❌ 不做内置待办列表

Pi 认为待办列表通常更多是混淆模型而非帮助。如果需要任务追踪，建议写入 TODO.md 文件，由代理自行读写。简单、可见、可版本控制。

❌ 不做内置规划模式

Pi 没有"规划模式"。让用户直接告诉代理思考问题而不修改文件就足够了。跨会话的持久规划写入 PLAN.md 文件。如果确实需要限制代理，可以使用 --tools read,grep,find,ls 限制工具集。

❌ 不支持 MCP

Pi 明确不支持 MCP（Model Context Protocol）。作者认为 MCP server 将大量工具描述注入上下文中（Playwright MCP 占用 13.7k tokens，Chrome DevTools MCP 占用 18k tokens），浪费昂贵的上下文窗口。替代方案：构建简单的 CLI 工具 + README 文件，代理按需读取 README 了解用法，通过 bash 直接调用。实现了渐进式上下文加载。

❌ 不支持后台 bash

Pi 的 bash 工具以同步模式运行，没有内置的后台进程管理。作者认为后台进程管理增加了不必要的复杂性，推荐使用 tmux 来管理长时间运行的进程（开发服务器、LLDB 调试会话等），既能获得完整的可观测性，又能与代理协作调试。

❌ 不支持子代理

Pi 没有专用的子代理工具。作者认为子代理通常是"没有提前规划"的信号。如果需要在会话内收集上下文，建议先单独开一个会话，生成一份上下文工件，再在下一个会话中使用。这样你能获得完整的可观测性和控制力。如果确实需要子代理，可以让 Pi 通过 bash 自我启动。

📊 可观测性优先

Pi 的设计原则之一：用户应该能检查与模型交互的每一个方面。没有黑盒子，没有隐藏的子代理。完整的会话格式文档化，易于后处理，便于构建替代 UI。

🔗 上下文工程

作者认为精确控制进入模型上下文的内容是获得更好输出的关键。现有工具在用户背后注入内容且不在 UI 中显示，使得上下文工程极为困难或不可能。Pi 的 AGENTS.md 系统让用户完全掌控上下文。

一句话总结 Pi 的设计哲学： 如果一个功能不是作者个人需要的，它就不会被实现。Pi 是有主见的工具，不试图讨好所有人。如果你需要不同的工作方式，作者鼓励你 fork 项目。

工作流程

# 1. 安装 Pi
npm install -g --ignore-scripts @earendil-works/pi-coding-agent

# 2. 在项目目录中启动
pi

# 3. 登录或设置 API Key
/login                         # 订阅式供应商
export ANTHROPIC_API_KEY=...   # 或直接设置环境变量

一个典型的 Pi 使用流程如下：

① 用户输入提示

用户在 TUI 编辑器中输入消息，支持模糊文件搜索、路径补全、拖放和多行粘贴。

② 代理循环启动

pi-agent-core 将系统提示 + AGENTS.md + 用户消息发送给 pi-ai，pi-ai 路由到当前配置的 LLM 供应商。

③ 工具执行

LLM 返回工具调用（read、write、edit、bash），代理执行并将结果反馈给 LLM，循环直到 LLM 产生最终文本响应。

④ 差分渲染

pi-tui 使用差分渲染算法，只更新终端中变化的部分，提供几乎无闪烁的体验。

⑤ 会话持久化

每次交互写入会话日志，支持 continue、resume、branching。可导出 HTML。

跨模型上下文传递

这是 Pi 最独特的功能之一——在会话中无缝切换 LLM 供应商：

import { getModel, complete, Context } from '@mariozechner/pi-ai';

// 先使用 Claude
const claude = getModel('anthropic', 'claude-sonnet-4-5');
const context: Context = { messages: [{ role: 'user', content: '25 × 18 = ?' }] };

const claudeResponse = await complete(claude, context, { thinkingEnabled: true });
context.messages.push(claudeResponse);

// 切换到 GPT —— Claude 的思维链转为 <thinking> 标签文本
const gpt = getModel('openai', 'gpt-5.1-codex');
context.messages.push({ role: 'user', content: '结果正确吗？' });
const gptResponse = await complete(gpt, context);

// 切换到 Gemini
const gemini = getModel('google', 'gemini-2.5-flash');
context.messages.push({ role: 'user', content: '刚才的问题是什么？' });
const geminiResponse = await complete(gemini, context);

// 序列化上下文为 JSON（存储或传输）
const serialized = JSON.stringify(context);
