🛠️ Superpowers & gstack 使用说明

两大社区框架，将 Claude Code 从通用助手升级为遵循工程纪律的虚拟开发团队。 本文涵盖核心概念、安装部署、工作流命令及对比选型建议。

📖 框架总览

🔵 Superpowers

定位：技能驱动的工程方法论，强制 TDD 与分阶段工作流，约束 AI 按规范行事。

作者：Jesse Vincent (obra) | 仓库：github.com/obra/superpowers

核心哲学：AI 是“热情但缺乏判断力的初级工程师”，必须通过纪律和流程来约束[reference:0]。

🟠 gstack

定位：角色驱动的虚拟工程团队，用 CEO、架构师、QA 等角色分工完善决策链。

作者：Garry Tan (YC CEO) | 仓库：github.com/garrytan/gstack

核心哲学：不要让一个 AI 扮演所有角色，而是组建一支结构化的专家团队各司其职[reference:1]。

一句话区别：Superpowers 关心的是 AI 该按照什么方法做事；gstack 关心的是 AI 该以什么角色做判断[reference:2]。

⚙️ 安装部署

两个框架均以 Claude Code Skills 形式存在，安装即启用。前置要求：已安装 Claude Code、Git，gstack 额外要求 Bun ≥ 1.0[reference:3]。

🔵 安装 Superpowers

通过 Claude Code 插件市场安装：

# 1. 添加官方插件源
/plugin marketplace add anthropic/plugins

# 2. 安装 Superpowers 插件
/plugin install superpowers

安装后，Superpowers 的技能会自动触发，无需手动调用[reference:4]。

验证：输入“帮我实现一个新功能”，观察是否自动触发 brainstorm → plan → TDD 流程[reference:5]。

🟠 安装 gstack

# 克隆仓库并运行 setup
git clone --single-branch --depth 1 \
  https://github.com/garrytan/gstack.git \
  ~/.claude/skills/gstack

cd ~/.claude/skills/gstack && ./setup

setup 脚本会自动检测已安装的编码代理（Claude Code、Cursor、Codex、Gemini CLI 等）并完成适配[reference:6]。

验证：输入 /help 检查 gstack 命令是否已生效[reference:7]。
⚠️ 注意：同时安装两个框架可能导致技能冲突[reference:8]。如需组合使用，请参考下方组合使用章节。

🔵 Superpowers 详解

Superpowers 的核心铁律：没有失败测试，就不写生产代码。它将开发过程拆解为 7 个强制阶段，每个阶段都有检查门禁，不可跳过[reference:9]。

🏗️ 七大核心工作流

阶段 技能 说明
1 Brainstorming Socratic 式追问，厘清需求边界，生成设计文档
2 Using Git Worktrees 自动创建隔离分支，验证 baseline 测试
3 Writing Plans 将需求拆成 2-5 分钟可验证的微任务
4 Subagent-Driven Development 并行子 Agent 执行 + 两阶段 Review（Spec 符合性 → 代码质量）
5 Test-Driven Development 强制 Red-Green-Refactor，先写失败测试再实现
6 Code Review 自动生成 Review 意见，关键问题直接 Block
7 Finishing a Branch 测试通过 → 合并/PR 建议 → 清理 Worktree

🎯 适用场景

  • 对代码质量要求高，担心 AI 跳过测试或规范流程
  • 需要 TDD 铁律提升代码覆盖率（社区反馈从 40% 提升到 95% 以上）[reference:10]
  • 希望 AI 先设计后实现，避免"凭感觉写代码"
核心理念：Superpowers 不是让 Agent 更聪明，而是让它必须遵守真实软件工程流程[reference:11]。

🟠 gstack 详解

gstack 将 Claude Code 转变为包含 20+ 角色的虚拟工程团队，遵循 Think → Plan → Build → Review → Test → Ship → Reflect 的 Sprint 结构[reference:12]。

👥 核心角色与命令

命令 角色 功能说明
/office-hours YC Partner 六道强制追问，重新定义产品方向。设计文档自动流入下游技能[reference:13]
/plan-ceo-review CEO / Founder 从第一性原理重新审视问题，四种模式：Expansion / Selective Expansion / Hold Scope / Reduction[reference:14]
/plan-eng-review Engineering Manager 锁定架构、数据流、边界情况、测试矩阵[reference:15]
/plan-design-review Senior Designer 评分设计维度（0-10），AI Slop 检测，交互式设计选择[reference:16]
/review Staff Engineer 发现 CI 通过但生产爆炸的 Bug，自动修复明显问题，标记完整性缺陷[reference:17]
/qa QA Lead 真实浏览器自动化测试，自动点击、登录、生成多尺寸截图，发现 Bug 后自动修复并验证[reference:18]
/ship Release Manager CI/CD 集成：自动合并、运行测试、打包推送、生成发布说明[reference:19]
/guard Security Officer 最高安全模式：危险操作前弹窗确认，锁定修改范围[reference:20]

🎯 适用场景

  • Solo Developer 或小团队，缺乏 PM、QA、安全等角色视角
  • 从 0 到 1 冲刺 MVP，需要完整的产品构思 → 发布闭环
  • 需要多维度的角色互相挑刺，补齐决策盲区[reference:21]
真实数据：Garry Tan 用 gstack 在 60 天内交付超过 60 万行生产代码（35% 为测试代码），平均每天 10,000-20,000 行[reference:22]。

⚖️ 对比与选型

维度 🔵 Superpowers 🟠 gstack
核心哲学 工程方法论（怎么做） 角色治理（谁来做判断）
架构模式 技能驱动，7 阶段强制工作流 角色驱动，20+ 虚拟专家团队
关键机制 强制 TDD + Subagent 执行 Sprint 结构 + 角色 Review 链
适用规模 个人到小团队 个人到小团队
学习曲线 中等 — 需先理解方法论 低 — 复制技能即可使用[reference:23]
治理机制 方法论强制执行纪律 无内置治理层，信任流程[reference:24]

🧭 选型决策指南

你的痛点 推荐工具 原因
怕 Agent 乱改、跳过 spec/test/review Superpowers 用纪律约束 Agent 行为，强制每一步规范流程[reference:25]
缺 PM、QA、安全等互相挑刺的视角 gstack 利用虚拟团队补齐决策盲区[reference:26]
日常主力工具，只装一个 Superpowers 覆盖开发全流程，纪律性强，社区最活跃[reference:27]
从 0 冲刺 MVP，产品构思到上线 gstack 完整的 Think → Ship 闭环，YC 验证的实战工作流[reference:28]
三层分工模型：gstack 做决策（判断做什么）、GSD 做上下文稳定（保证不跑偏）、Superpowers 做执行（方法论闭环）[reference:29]。

🔗 组合使用

两个框架可以互补协作：gstack 负责决策层（做什么），Superpowers 负责执行层（怎么做）[reference:30]。

📋 推荐全链路闭环流程

阶段 工具 命令 产出
需求围攻 gstack /office-hours 产品设计文档，理清核心逻辑[reference:31]
深度脑暴 Superpowers /superpowers:brainstorm DESIGN.md，覆盖边界情况[reference:32]
角色审查 gstack /plan-ceo-review
/plan-eng-review		 CEO + 架构师多维审查通过
任务拆解 Superpowers /superpowers:write-plan 2-5 分钟的微任务列表
强制 TDD 执行 Superpowers /superpowers:execute-plan 先测试后代码，子 Agent 并行执行[reference:33]

⚠️ 关键提醒：不要盲目安装全部技能。组合使用的核心是根据当前实际短板选择性地部署，理解每个层的作用后再组装属于你的工作流[reference:34]。
